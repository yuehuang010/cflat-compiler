# vectorize keyword (enforced auto-vectorization)

`vectorize` keyword LANDED 2026-06-06 (first HPC feature from [[project-hpc-language-features]] Tier 1). Prefix modifier on a counted loop: `vectorize for (...)` / `vectorize while (...)`. Contract = "this loop is vectorized at -O2 or the compile hard-errors." Decisions (from design session): hard-error always, **enforced only at -O2** (no-op below - the vectorizer only runs at O2), FP reductions rejected for now, range+while both allowed.

**CONTRACT TIGHTENED 2026-06-09 (Detection B): a surviving runtime alias check is now a FAILURE.** The original contract ACCEPTED a versioned loop (one LLVM guards with a `vector.memcheck` runtime overlap test) as success. That was reversed: `vectorize` now means the CLEAN, unconditional vector loop - if LLVM had to version the loop because it could not prove the pointers disjoint, the compile hard-errors. Rationale (user, high-risk/reward for a keyword-driven optimization): the whole point of `vectorize` over a hot kernel is the memcheck-free loop; silently keeping the runtime check defeats it. This is a GENERAL tightening - it fires on raw `int*` may-alias loops too (message: "pass the buffers as `span<T>`/`T[]` (noalias) ..."), not just spans. SAFE because no core library uses the `vectorize` keyword (all core mentions are comments), so only user code that explicitly opts in is affected.

**Detection-B mechanism (LLVMBackend.cpp OptimizeModule scan):** a vectorized loop retains BOTH `cflat.vectorize.line` AND `llvm.loop.isvectorized` (LLVM copies our custom metadata onto the vectorized loop). So the scan no longer just `continue`s on `isvectorized`: for a vectorized tagged loop it builds a per-function `llvm::DominatorTree` (lazy, `#include <llvm/IR/Dominators.h>`) and walks the latch's idom chain for a block whose name `.contains("memcheck")`. The vector body is reached only when the runtime check passes, so a `vector.memcheck` block DOMINATES it; a clean vector loop has none -> no false positive (and dominance attributes correctly even when a function has both a clean and a versioned loop). Message via `ComposeMemcheckFailure` with prefix "vectorize loop did not vectorize cleanly:" (vs "could not be vectorized:" for the not-vectorized-at-all case). Name-based memcheck detection is the same idiom the test harness (span_noalias_check.cb) already uses; if LLVM renames the block, detection soft-fails (loops that should error pass) - not a crash.

**Detection A (better hints, same change):** `VectorizeLoopInfo` gained `hasSpanAccessor`/`spanAccessor`/`spanReceiver`/`spanLine`/`spanCol`. Recorded TYPE-AWARE at codegen (not the syntactic FindFirstCall scan, which can't tell a span from a list) when a span `get`/`set`/rvalue-`[]` is lowered inside a vectorize body: a `MainListener::currentVectorizeBodyLine_` marker (saved/restored around the while/for body emit, since `vectorizeActive_` is consumed too early) gates `LLVMBackend::NoteVectorizeSpanAccessor(line, accessor, receiver, ...)`. Span recognition = `ArrayViewBufferFieldIndex(typeName) >= 0` (a struct with an `IsArrayView` field named `_ptr`; excludes may-alias `view<T>`). When B fires and A recorded an accessor, the message NAMES it and prescribes `T[] v = <recv>.data();`. lvalue `y[i]` is lowered to the noalias field access (see [[span-view]]) so it does NOT record - only the genuinely-slow accessors do. Tests: Test/vectorize/neg_span_getset.cb (+ `:expect_fail_msg` harness helper in test_hpc.bat).

> GOTCHA burned during impl: `GetDataStructure` returns `StructData` BY VALUE. `GetDataStructure(name).StructFields[i]` binds a `const&` through `vector::operator[]` (a function call), which DEFEATS C++ lifetime extension -> the temporary dies at the full-expression end and the ref dangles (manifested as "unknown type ''" from an empty TypeName). Bind the StructData to a named `const auto&` FIRST (direct binding does extend lifetime), then index it.

**4-site implementation** (mirrors the soft-keyword pattern):
1. `CFlat.g4`: `vectorizeStatement : 'vectorize' iterationStatement ;` added to `statement`. Inline-literal soft keyword (like lock/program).
2. `MainListener.h ParseStatement`: retrieve `vectorizeStatement()`, reject do-while/foreach `In` up front, set `vectorizeActive_`/`vectorizeLine_`, reuse the existing while/for emit. Consume the flag at the top of the iteration branch (so nested loops don't inherit). After the back-edge branch (`CreateContinueCall` for while, `CreateBlockBreak(blockCondition)` for for), call `compiler->AttachVectorizeHintToCurrentLatch(line)`.
3. `LLVMBackend.h AttachVectorizeHintToCurrentLatch`: stamps `!llvm.loop` = distinct {self, !{"llvm.loop.vectorize.enable", i1 true}, !{"cflat.vectorize.line", i32 <line>}} on the latch terminator; increments `vectorizeLoopCount_`.
4. `LLVMBackend.cpp OptimizeModule`: enforcement.

**Enforcement = IR scan, NOT diagnostics** (the hard-won part). Forcing `vectorize.enable=true` makes a failed loop survive optimization still carrying the force hint WITHOUT `llvm.loop.isvectorized` - exactly what LLVM's own `WarnMissedTransformations` keys on. So post-`MPM.run`, walk every block's terminator: a loop ID carrying `cflat.vectorize.line` but NOT `llvm.loop.isvectorized` = FAILED -> read line from metadata, `LogError`. A vectorized loop drops the custom tag / gains isvectorized; an eliminated loop has no metadata (vacuously fine). The pass/fail DECISION is the IR scan; the failure diagnostic came from pass `transform-warning` (DK_OptimizationFailure, generic), NOT `loop-vectorize` - that's why a pass-name filter for the reason failed.

**Feedback improvement (LANDED 2026-06-06, same day):** failure messages now name a specific, located cause via two inputs combined in `ComposeVectorizeFailure`: (1) LLVM's `loop-vectorize` ANALYSIS remark - captured by keying on the `"loop not vectorized:"` MESSAGE prefix (the remark's PASS NAME is often EMPTY for the marked loop, so a pass=="loop-vectorize" filter silently dropped them - the bug), then translated by the stable remark NAME (`CantVectorizeLibcall`, `NonReductionValueUsedOutsideLoop`, ...); and (2) CFlat-side AST facts gathered at codegen into `LLVMBackend::VectorizeLoopInfo` (per-loop, matched by line): is-while, while-condition shape (`ConditionIsSentinel` = has ==/!= but no </<=/>/>= via `ScanComparisons` over RelationalExpression/EqualityExpression contexts), and the first body call (`FindFirstCall` = first PostfixExpression with a non-empty `argumentExpressionList`). The message POINTS at the offending construct (call site col, condition col), not just the loop. Results: neg_call -> "the loop body calls 'printf', which was not inlined..." at the call; neg_while -> "no countable trip count; the condition 'p!=nullptr' is not a counted comparison..." at the condition (overrides LLVM's misleading "not a reduction"); neg_carried_dep (LLVM emits NO loop-vectorize remark for it, only the generic transform-warning) -> structural fallback "most often a loop-carried dependence, e.g. a[k]=a[k-1]+...". Design choice: AST facts ENRICH the failure message (used only when the IR scan already flagged failure), they do NOT pre-reject - avoids false-rejecting valid loops (e.g. a countable `while (i != n)`).

DEFERRED (user "good enough for now, revisit later" 2026-06-06): the carried-dependence message is a structural GUESS ("most often a loop-carried dependence") because LLVM emits no loop-vectorize remark for that shape and we do no subscript analysis. Future "true dependence detection" = read the body's array subscripts and prove the dependence. Three levels: (1) today = guess; (2) cheap pattern heuristic = detect `base[i +/- k]` read + `base[i]` write on the same name, name the subscripts+locations; (3) full affine analysis = induction-var detection + affine index normalization + same-base/aliasing + distance test = a mini LoopAccessAnalysis. Level 3 IS the noalias/safelen feature ([[project-noalias-design]]) wearing a diagnostics hat - do it THERE, not as a separate diag bolt-on. Level 2 is the pragmatic middle if only better carried-dep messages are wanted.

**ROOT CAUSE that blocked everything (broad win): the optimizer's PassBuilder had no TargetMachine.** `OptimizeModule` built `llvm::PassBuilder PB;` with no TM -> no TargetTransformInfo -> the loop vectorizer can't cost vector instructions and **silently declines to vectorize almost everything** (only memcpy/memset idioms + SLP still fire; that's why a `c[k]=a[k]` "worked" - it became a memcpy idiom). The TM was only created later in `EmitExecutable` for object emission. Fix: new `CreateOptTargetMachine()` (resolves triple from platformValue + cpu from targetCpu_), `PassBuilder PB(TM.get(), pto)` with PTO.LoopVectorization/SLPVectorization=true, and `module->setDataLayout(TM->createDataLayout())` before opt. **This makes ALL -O2 compiles actually vectorize/optimize for the target now, not just vectorize loops - a latent perf improvement across the compiler.** When stuck on "textbook loop won't vectorize," CHECK FOR A TARGETMACHINE before blaming -cpu/cost-model/legality.

**Fixed-array double-alloca bug found + fixed (pre-existing, severe).** `int[N] a` emitted `alloca [N x i32], iX N` - size applied twice (once in the `[N x T]` element type from GetType, once as the alloca element-count) = N x N over-allocation (int[1024] -> 4MB not 4KB; stack-overflow risk AND it defeated the vectorizer's bounds analysis). Fix: in `LLVMBackend.h CreateLocalVariable`, `if (typeValue.ConstArraySize > 0) arraySize = nullptr;` (the dimension is already in the type). One chokepoint, covers all callers, multi-dim safe.

**Also: `--out-lli` now writes AFTER optimization** (user-authorized). WriteIR block moved below `OptimizeModule` in `Compile`, so the dumped IR reflects the final vectorized/inlined module instead of pre-opt. At -O0 it now shows post-baseline-pass IR (SROA/mem2reg ran). DI still survives (verified).

**Tests**: `Test/test_vectorize.cb` (top-level, picked up by test.bat at -O0 where vectorize is a no-op - validates the loops run correctly). `test_vectorize.bat` (NEW, the real -O2 contract harness, mirrors test_debug_info.bat): positive must compile+run at -O2; `Test/vectorize/neg_*.cb` (carried-dep via pointer param, non-inlinable printf call, non-countable while pointer-chase) must each fail with "could not be vectorized". NEGATIVES MUST USE OPAQUE DATA (pointer params) - a local-array negative gets constant-folded away and the loop vanishes (false pass). Negatives live in `Test/vectorize/` (a subdir, NOT globbed by test.bat's `Test\test_*.cb` nor test_err.bat) and CAN'T use `expect_error` (the vectorize error fires in OptimizeModule, after the AST-walk scope-exit check). All suites green: test.bat, test_vectorize.bat (both configs), test_lsp.bat (101).
