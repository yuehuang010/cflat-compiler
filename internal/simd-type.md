**PERF GOTCHA - `simd<T,N>.fma` is a TRAP on the default target**: cflat's CreateOptTargetMachine targets generic x86-64 (NO FMA unit; the "--cpu native -> generic" papercut). `llvm.fma` is single-rounded so LLVM CANNOT fold it to fmulpd+faddpd - it emits a PER-LANE libm `fma()` CALL. Measured in nbody force kernel: `.fma(dx,s,acc)` = 19 GFLOP/s vs plain `acc + dx*s` = 46 GFLOP/s (>2x slower). Use `.fma` ONLY when you need single-rounding precision AND know the target has FMA; for plain accumulate write `acc+a*b`. example/hpc/nbody.cb force kernel is now FULL explicit SIMD (simd<double,8>.load SoA + one/simd.sqrt(r2) + vector mul/add, NO fma) at PARITY with the old `vectorize for` (~46 vs ~44 GFLOP/s fair same-machine; the eval's stale "81" was a cooler-run number on the heterogeneous Zen5 box [[project-host-cpu-cache-topology]]). doc/HPC.md has an FMA caveat. REAL FIX (future) = target host features (+fma,+avx2) by default -> makes .fma fast AND widens all auto-vec. NOTE width: simd<double,8> single-accumulator was at parity; going wider/unrolling did NOT help (sqrt/div throughput bound).

Design session 2026-06-06 for `simd<T,N>`, the explicit-SIMD value type. Picked as the next HPC feature after [[project-vectorize-keyword]] landed (Tier 2 escape hatch from [[project-hpc-language-features]]).

**What it is**: a fixed-width SIMD value - N lanes of element type T, lowered to LLVM `FixedVectorType` `<N x T>`. The *guaranteed* counterpart to `vectorize`'s *best-effort* auto-vec: when a `vectorize` loop hard-errors (carried dep, shuffle, horizontal reduce), simd is the escape hatch.

**Category - it is a PRIMITIVE, not a struct/template (both clarified this session):**
- NOT a struct/aggregate. struct=`{...}` and array=`[N x T]` are aggregates in memory; simd=`<N x T>` is a VECTOR primitive in a register. Same category as `int`/`float`, NOT `Point`/`list<T>`. If it were a struct, LLVM would never vectorize arithmetic (N scalar ops) - defeats the purpose. `v[i]`=extractelement/insertelement (register), NOT GEP+load (memory). No destructor, no ownership, no move, no pointer, no interface. Small blast radius - lands like `function<>`, NOT like `string`.
- NOT a real generic/template. It's a builtin SPECIAL-FORM type recognized directly in `ParseDeclarationSpecifiers` (no `.cb` definition, no monomorphization, no `simd__float8` symbol). Precedent = `function<R(P)>` (CFlat.g4:299-302), the ONLY existing angle-bracket builtin special-form. Tuple `(A,B)` is a paren builtin; `string` is the cautionary keyword+core hybrid with ~20+ `TypeName=="string"` branches. simd is the 2nd `<...>` special-form. Can't be a real generic anyway: N is an INTEGER (value), and CFlat generics have no integer/non-type template params.

**Lowering = LLVM IR, not direct CPU instructions**: emit `<8 x float>` + vector `fadd`/`fmul`/etc.; LLVM's TargetMachine (the one CreateOptTargetMachine added in the vectorize work) legalizes to `vmulps`/SSE-split per target. Stay portable - use plain IR ops + target-INDEPENDENT intrinsics (`shufflevector`, `@llvm.vector.reduce.*`), NOT `@llvm.x86.*`.

**LOCKED decisions:**
- T = numeric scalars only (i8..i64, u8..u64, float, double). Non-numeric T -> LogError.
- N = compile-time integer literal, **must be power of 2** (2..64). **Non-power-of-2 = COMPILE ERROR** (confirmed 2026-06-06) with hint "did you mean simd<float,8>?". Rationale: follows Rust SupportedLaneCount (C++ std::simd allows any N); simd is the explicit hw-control hatch so silent 7->8 pad (LLVM widens to 8-wide reg, lane 7 dead) contradicts intent; erroring is non-breaking to relax later, "any N then tighten" would break code. NOTE: `<7 x float>` IS legal LLVM IR and stays 7 lanes (v[7] OOB) - the "8" is only the hardware register, not usable capacity. We choose to reject it at the language level.
- Value semantics, by value, no ownership interaction.
- v1 scope: splat (scalar->all lanes via CreateVectorSplat), element-list init, element-wise `+ - * /`, comparisons (->`<N x i1>`, surface as `simd<i32,N>` mask via SExt), subscript read/write, load/store vs `T*`/`T[N]`. DEFER shuffles/gather/scatter/masking/horizontal-reduce to v2.

**Implementation plan (6 phases, grounded in code):**
- P0 Representation: add `bool IsSimd; uint64_t SimdLanes;` to TypeAndValue (LLVMBackend.h:196). **MUST mirror in all 4 JSON serialize sites** like ConstArraySize does (LLVMBackend.cpp:1961/2006, LLVMBackend.h:9076/9105) or bitcode/--init cache + LSP index silently mistype. `IsInteger()/IsFloatingPoint()/IsTypeMatch()` must stay opaque to simd (return -1) so scalar coercion skips it - "don't let simd become another string".
- P1 Grammar: `simdType : 'simd' '<' typeSpecifier ',' DecimalLiteral '>'` as alt in typeSpecifier (CFlat.g4:256), mirror `functionPointerSpecifier`. `simd` = inline-literal soft keyword.
- P2 Type parse: detect in BOTH ParseDeclarationSpecifiers copies (MainListener.h ~330 and ~1390, like func-ptr handling), validate, set IsSimd/SimdLanes.
- P3 Lowering: branch in GetType BEFORE the ConstArraySize branch (LLVMBackend.h:6880) -> `FixedVectorType::get(elemTy, N)`.
- P4 Construct + operators: arithmetic is mostly FREE - LLVM's CreateFAdd/CreateAdd/CreateICmp (the switch at LLVMBackend.h:5945-6030) are ALREADY element-wise on vector operands. Real work = (a) bypass the ptr-arith/bitwise branch (5905-5939, scalar-int only) when either side isVectorTy(); (b) fix `Upconvert(left,right)` (5946-5947, scalar-only) to require matching `<N x T>` or splat a scalar operand, error on lane/elem mismatch. Splat hooks the ConstArraySize value-init path (MainListener.h:4029+).
- P5 Subscript + load/store bridge (extractelement/insertelement; aligned `load <N x T>`).
- P6 Tests: Test/test_simd.cb (top-level, runs at -O0 - vector IR executes under lli, no -O2 needed unlike vectorize); Test/errors/err_simd_*.cb via expect_error (non-numeric T, non-pow2 N, lane mismatch - these fire at codegen so expect_error WORKS, unlike vectorize negs); doc/LANGUAGE.md section; run test.bat + test_lsp.bat (P0 touches LSP index).
