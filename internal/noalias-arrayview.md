The noalias feature ([[project-hpc-language-features]] Tier 1, the escape-hatch counterpart that [[project-vectorize-keyword]]'s diagnostics point you toward) landed on a clean, small design: a distinct **thin `int[]` array-view type**, NOT a `noalias` keyword qualifier and NOT noalias-by-default.

**THE LOCKED DESIGN:**
- **`int[]` is a distinct type = `int*` repr + a noalias contract.** In C, `int[]` param decays to `int*` (a lie - same type). CFlat makes the lie true the other way: `int[]` is a *stronger, distinct* type; `int*` stays the raw, may-alias, arithmetic-capable C pointer. Noalias is "by SPELLING" not "by default" - you migrate a hot kernel into `int[]` deliberately; existing `int*` code is never silently reinterpreted (avoids the Fortran silent-UB footgun). Mnemonic: brackets = "a buffer, indexed not walked."
- **THIN, not fat.** Same LLVM pointer repr as `int*` + one flag `IsArrayView` on TypeAndValue. NO length carried. User tracks length manually (e.g. `void saxpy(int n, float[] x, float[] y)`) - DECISION 2026-06-06, user wants the length VISIBLE in the signature, not a hidden fat-pointer field. Decomposes to the same inner-loop codegen as Rust's fat `&[T]` anyway (Rust splits {ptr,len} into 2 scalar regs at the ABI; the hot loop runs on the thin half + trip count) - so thin loses nothing in the loop, only the bundled-length convenience, and gains zero DST machinery + no extra call-site register.
- **Arithmetic BANNED on `int[]`, indexing `a[i]` allowed.** Type-driven, so the bump allocators (entry/arena/bucket/block) keep their `int*` + `*p++` idiom UNTOUCHED - no global arithmetic ban needed.
- **Conversion lattice: `int[] -> int*` IMPLICIT decay (lose the guarantee, always safe; reuses the existing implicit-upcast path - `int[]` is to `int*` as derived is to base). `int* -> int[]` IMPOSSIBLE - NO cast, NO escape hatch** (user dropped the `(int[])p` cast 2026-06-06). This is the soundness keystone (below).
- **Overload identity**: `f(float[])` and `f(float*)` are distinct signatures; `float[]` arg matches `float*` param (decay), never reverse. Asymmetric, like upcast.
- **C-ABI fence**: `int[]` passed to extern C decays to `int*` and drops noalias first (C aliases freely). Falls out of the decay rule + existing rule that ExtractCSignatures/CompileCHeader stamp C ptr params as raw.
- **Carve-out is automatic**: atomic/mutex/channel internals (shared-mutable-through-ptr, the Rust UnsafeCell case) just stay `int*`-flavored, never adopt `int[]`.

**THE SOUNDNESS KEYSTONE (why no-cast + arithmetic-ban is provable, not trusted):** with no cast AND no arithmetic, you CANNOT construct an `int[]` pointing partway into an allocation (`a+5` banned; `&a[5]` is an `int*` and can't become `int[]`). So **every `int[]` is a WHOLE-allocation view.** Therefore two `int[]` alias IFF same base pointer = plain pointer equality, NO extent needed - which a thin view CAN check. This RESTORES the full safety net that thin-with-cast had given up (the earlier worry was offset sub-views needing a length to range-check; they're now unconstructible). Exhaustive `int[]` construction set (the soundness boundary): `new int[n]` (blessed heap ctor, types as `int[]` directly), `T[N]` fixed-array decay, an `int[]` param, `int[] b = a` copy (same base), an `array<T>`/`list<T>` whole-buffer accessor. Residual hazard collapses to one shape - `int[] b=a; f(a,b)` or `f(a,a)` (two views, same allocation) - caught by a **Debug-build base-pointer-equality check among noalias args at the call site** (cheap, complete, Debug-only, NOT in the hot loop, compiled out of Release).

**FROZEN FOR THE FUTURE - `a[i:j]` slicing is the seam that reopens everything.** A sub-slice operator manufactures an `int[]` with base partway in (different base, overlapping range) - breaks the whole-allocation invariant, "distinct base => disjoint" becomes WRONG, and you'd need the fat repr + a real range-overlap check. Rust supports slicing precisely because it pays BOTH: fat `{ptr,len}` + a borrow checker (rejects `f(&mut a[0..5], &mut a[3..8])` at compile time, emits LLVM noalias only because disjointness is already proven). CFlat sidesteps the proof by FORBIDDING the constructs that create overlap, price = no slicing. Do NOT add `a[i:j]` casually - it signs you up for Rust's bill (fat views + borrow-checker-grade analysis or runtime range check).

**HOW IT INTERACTS WITH SIMD/vectorize (the actual win):** noalias is a COMPILE-TIME attribute whose purpose is to DELETE a RUNTIME check the vectorizer would otherwise insert. Today `int*` params still vectorize at -O2 but via **loop versioning**: LLVM emits a runtime bounds-overlap test + a vector path + a scalar fallback (correct, but pays a branch, code bloat, pessimizes small n; bails to scalar if it can't bound the ranges). Emitting `noalias` on `int[]` params -> AA returns NoAlias -> vectorizer drops the runtime check AND the scalar fallback -> unconditional clean SIMD. So `int[]` is the canonical fix a failed `vectorize for` diagnostic points at. RELEASE: noalias trusted, no check, max speed, **silent wrong-numbers if the contract is violated** (Fortran footgun, narrowed) -> THAT is why the Debug call-site check exists (catch the lie in testing before Release trusts it). Unifying note: LLVM's loop-versioning check IS the range-overlap test (extents from trip count) - the same one slicing would need; whole-buffer `int[]`+noalias skips it entirely. `simd<T,N>` (explicit) doesn't depend on AA to vectorize, but noalias still helps it reorder/coalesce loads/stores (moot in v1 - simd load/store deferred).

**SURVEY CONTEXT (the reasoning that produced the design, condensed):**
Enforcement lanes: C=may-alias default (safe/slow). Fortran=noalias default TRUSTED-not-checked (silent UB, the footgun BLAS accepted; every compiler ships -fno-aliasing). Rust=noalias (`&mut`) compiler-CHECKED via borrow checker (sound; cost=friction+learning cliff; UnsafeCell carve-out for interior mutability; is LLVM's noalias-miscompile stress test - WARNING if CFlat emits aggressively). **CFlat 4th lane** = the thin-`int[]` design above: provable (no-cast+arith-ban make whole-allocation the only constructible form) + Debug overlap check for the residual = cheaper than Rust, safer than Fortran, fits the "LLVM assert -> real compiler error" rule.
Boolean alias/noalias is the WRONG abstraction for a vectorizer - it wants a DISTANCE (safelen), not a predicate: noalias=dist-inf(any width), may-alias=dist-0(scalar), middle="conflicts no closer than d"->vectorize to width d (= OpenMP `safelen`/Intel ivdep, > C restrict). RESOLUTION: the `int[]` TYPE carries the binary disjointness between DISTINCT buffers (the 90% BLAS case); the DISTANCE/safelen middle (self-dependence within ONE buffer, stencils `a[i]=a[i-1]+..`) is a LOOP property -> a future loop annotation, NOT this type. Keep distance(algorithm, target-independent) separate from width(machine: x86 compile-const, ARM-SVE/RISCV-RVV runtime vscale) - never bake hardware width into source; backend does min(safelen,width). So this feature handles the headline case via types and demotes distance to a narrower later loop-level refinement.
